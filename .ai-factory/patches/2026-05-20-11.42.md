# Tags endpoint fallback + claude -p code-fence stripping

**Date:** 2026-05-20 11:42
**Files:** src/tracker_assistant/adapters/timetta_adapter.py, scripts/enrich_task.py, tests/test_fixes.py
**Severity:** medium

## Problem

Two bugs caused `enrich_task.py` to fail:

1. `GET /Tags` returned 404 and logged ERROR, enrich_task received empty tags list.
2. `claude -p` wrapped its JSON output in markdown code fences (` ```json ... ``` `), causing `json.loads` to fail with "Expecting value: line 1 column 1".

## Root Cause

1. The OData endpoint name for tags varies between Timetta instances/versions. The skill reference documented `/Tags` but the actual instance uses a different path. `_request` also always logged at ERROR level for any HTTPError — making "expected" 404 probes noisy.
2. `claude -p` without `--output-format json` flag returns human-readable output by default, which may include markdown formatting including code fences around JSON blocks.

## Solution

**Tags:** Added `quiet: bool = False` parameter to `_request` — when `True`, 404 errors are logged at DEBUG instead of ERROR. `get_tags()` now iterates `_TAG_CANDIDATES = ["/Tags", "/WorkItemTypes", "/IssueCategories", "/Labels"]` using `quiet=True` and returns the first successful response. Logs `[FIX]` info when tags are found at a non-default path.

**Code fences:** In `_call_claude`, after getting stdout, if output starts with ```` ``` ````, strips the opening fence line and trailing ```` ``` ```` line before calling `json.loads`.

## Prevention

- When calling external CLI tools that may return markdown-formatted output, always strip code fences before JSON parsing.
- For OData endpoint discovery, use `quiet=True` probes or a candidate-list strategy rather than hard-coding a single path.
- Use `claude -p --output-format json` (if available) to force machine-readable output.
