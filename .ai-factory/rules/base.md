# Базовые правила проекта

> Автоматически определённые конвенции из анализа кодовой базы. Редактируйте по необходимости.

## Соглашения об именовании

- Файлы: `snake_case` (например, `task_cli.py`, `io_utils.py`, `timetta_adapter.py`)
- Переменные и функции: `snake_case` (например, `list_projects`, `create_task`, `task_id`)
- Классы: `PascalCase` (например, `Task`, `TimettaAdapter`)
- Константы: `UPPER_SNAKE_CASE` (например, `ROOT`, `SRC`)
- Приватные методы/переменные: `_prefixed_snake_case`

## Структура модулей

- `scripts/` — точки входа CLI (не импортируются как библиотека)
- `src/tracker_assistant/` — библиотечный код (публичный API через `__init__.py`)
- `src/tracker_assistant/adapters/` — адаптеры внешних API
- `tests/` — тесты (только pytest, без вложенных пакетов)
- `templates/` — JSON-шаблоны для CLI-операций
- `docs/` — документация

## Обработка ошибок

- `raise SystemExit("ERROR: ...")` — для фатальных ошибок конфигурации (токен не задан, файл не найден)
- `logging.warning(...)` — для мягких сбоев и нештатных ситуаций
- Исключения не перехватываются в pipeline-слое, всплывают на CLI-уровень
- Нет кастомных классов исключений — используется стандартная иерархия Python

## Логирование

- `logging.getLogger(__name__)` в каждом модуле
- DEBUG: вход в метод, HTTP-запросы, промежуточные данные
- INFO: итоговые результаты (настраивается через `--log-level`)
- WARNING: мягкие сбои, пустые операции
- Формат: `%(asctime)s %(levelname)s %(name)s %(message)s`

## Тестирование

- Фреймворк: pytest
- Расположение: `tests/test_*.py`
- Нет БД, нет внешних зависимостей в тестах — используются mock-объекты
- Запуск: `uv run pytest` или `uv run python -m pytest`

## Стиль кода

- `from __future__ import annotations` в начале файлов с аннотациями типов
- Аннотации типов обязательны для публичных функций
- Строки документации не используются — имена функций говорят сами за себя
- Комментарии только когда «почему» неочевидно
- Максимум 500 строк на файл
